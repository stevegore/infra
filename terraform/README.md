# Terraform — OCI homelab

Manages every resource in the `main` compartment (plus the IAM dynamic-group
and policy at the tenancy level). **OCI Resource Manager owns the state**;
local CLI is for fast plans only.

## Why no `backend.tf`?

ORM's job runner doesn't initialize custom backend blocks (s3, etc.) before
running `plan`/`apply`, even though the OCI docs imply otherwise — see commit
history for the failed attempt. Sticking to ORM-managed state keeps a single
source of truth and avoids the workaround.

## Workflow

```
edit locally → plan locally → push to GitHub → ORM job applies
```

- `terraform plan` is run **anywhere** that can pull the ORM stack's state.
- `terraform apply` runs **only via ORM**, so every change has an audit-trail
  job, an approval workflow, and an automatic state snapshot on the ORM side.

## First-time local setup

```bash
# 1. Source the env wrapper. It pulls the ORM-managed state into a local
#    cache file and drops a backend_override.tf that points at the cache.
source ../scripts/tf-env.sh

# 2. Re-init (the override changes the backend block).
terraform init -reconfigure

# 3. Plan.
terraform plan
```

Re-source `tf-env.sh` whenever you want a fresh state snapshot.

## Layout

| Path | Purpose |
| --- | --- |
| `versions.tf` | Terraform + provider version pins |
| `providers.tf` | OCI provider config (auth via `~/.oci/config` locally, ORM-injected in jobs) |
| `variables.tf` | Tenancy/compartment/region — defaults baked in so both ORM and local work without tfvars |
| `core.tf`, `kms.tf`, `identity.tf`, `object_storage.tf` | Auto-generated by ORM Resource Discovery (2026-05-24). Trimmed to the resources we want under TF control. Don't hand-edit until they've been refactored. |
| `vars-imported.tf` | Discovery-emitted static defaults (fault-domain, AD name, image OCID) — will fold into resource bodies during the refactor pass |
